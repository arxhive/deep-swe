# Phase 0 Research: Workflow Bench

This document records the technical decisions that ground the plan. Each entry follows: Decision, Rationale, Alternatives considered. Findings are based on direct inspection of the corpus, the task verifier scripts, the owner's Claude config, and the host environment in this worktree.

## R1. Language, packaging, and dependency policy

- **Decision**: Python 3.11+ in a single `uv`-managed package under `bench/` (`src/` layout), console entry point `wfbench` (`uv run wfbench ...`). Runtime uses only the standard library; `pytest` is the sole dev dependency.
- **Rationale**: Matches the deep-swe / Harbor / Pier ecosystem and the owner's installed `uv` (0.8.22). `tomllib` (stdlib since 3.11) parses `task.toml` natively, so no TOML dependency is needed. `subprocess` drives `docker` and `claude`. Honors NFR-007 (lean footprint). `src/` layout keeps the import package unambiguous and test discovery clean.
- **Alternatives considered**: (a) Reuse/fork Pier - rejected by C-001 (must be standalone, no Pier dependency). (b) Add `click`/`typer` for the CLI - rejected; `argparse` covers the small command surface without a dependency. (c) Add a docker SDK (`docker` Python package) - rejected; the host already drives docker via CLI and the SDK does not honor the platform-injection nuance, so a thin subprocess wrapper is simpler and more transparent.

## R2. The docker `--platform linux/amd64` gotcha (highest-risk detail)

- **Decision**: The harness MUST pass `--platform linux/amd64` explicitly on every `docker run`, `docker build`, and `docker pull`. A single `docker_cli.py` wrapper centralizes invocation and injects the flag for exactly those subcommands.
- **Rationale**: Verified on the host: the interactive shell defines a zsh function `docker()` that injects `--platform linux/amd64` only for `run`/`build` when `uname -m == arm64`. `subprocess` does NOT use shell functions - it execs `/usr/local/bin/docker` directly - so that injection is bypassed entirely. Task images are `linux/amd64` (e.g. `public.ecr.aws/d3j8x8q7/swe-bench-202605:<ext_id>`); the host is arm64 macOS. Without the explicit flag, `docker run` on arm64 would attempt the wrong architecture or warn/fail. Note: `docker exec`, `docker network`, `docker cp`, and `docker rm` do NOT accept `--platform`; the wrapper injects it only for `run`/`build`/`pull`.
- **Alternatives considered**: (a) Rely on the shell function via `shell=True` - rejected; fragile, shell-dependent, and a security/quoting risk. (b) Set `DOCKER_DEFAULT_PLATFORM=linux/amd64` in the subprocess env - viable as defense-in-depth and will be set as well, but the explicit per-command flag is the authoritative, self-documenting guarantee and the plan treats the flag as mandatory regardless.

## R3. Cross-architecture Claude runtime: build once, cache, reuse (C-007)

- **Decision**: Provision a `linux/amd64` Claude Code runtime exactly once and cache it on the host, then mount it read-only into every task container. Build mechanism: a one-time `docker run --platform linux/amd64 -v <cache>/runtime:/out node:20-bookworm-slim bash -c "npm i -g --prefix /out/npm @anthropic-ai/claude-code@<PINNED_VERSION> && cp -aL $(command -v node) /out/node"`. At task time, mount `<cache>/runtime` read-only at `/opt/wfbench` and prepend `/opt/wfbench/node` (dir of the node binary) and `/opt/wfbench/npm/bin` to `PATH`.
- **Rationale**: `@anthropic-ai/claude-code` bundles platform-specific native binaries (e.g. `ripgrep`), so the macOS-host install cannot run inside `linux/amd64` containers - it must be installed inside Linux. Building once and reusing avoids reinstalling for all 113 tasks (the only deliberate performance measure). A pinned version keeps comparisons reproducible across runs (NFR-001 spirit). Task images are debian/ubuntu-family glibc bases (verified base `mars-base` is debian-derived via `apt`/`git`), so a `node:20-bookworm-slim` (glibc) runtime is ABI-compatible.
- **Risks / open assumptions**:
  - **glibc compatibility (R-RISK-1)**: assumes all task images are glibc-based. If any image is musl-based (Alpine), the bundled binaries would fail. Mitigation: surface a clear runtime error if `claude --version` fails inside a task container during the first task, and record the task as `errored` (not a silent pass). The corpus sample inspected uses a debian base; full enumeration is deferred to implementation but does not change the design.
  - **node major version (R-RISK-2)**: `claude-code` requires Node 18+. `node:20-bookworm-slim` satisfies this. The pinned claude-code version and node 20 are recorded in `config.py` constants.
- **Alternatives considered**: (a) Copy the macOS host's installed claude-code into the container - rejected; wrong-architecture native binaries. (b) `npm i -g` inside each task container at run time - rejected; requires network during the agent phase per task and wastes time/bandwidth across 113 tasks; also some images may lack npm. (c) Bake a dedicated runtime image and `docker pull`/build it - heavier; a mounted directory is simpler and avoids publishing an image, though a future optimization could switch to an image without changing the harness contract.

## R4. Owner Claude config: writable resolved copy, symlink-safe, secrets excluded (C-009, FR-008)

- **Decision**: For each run, build a writable copy of the owner's resolved `~/.claude` into the run dir and mount it at the container HOME's `.claude`. Copy with symlink resolution (`shutil.copytree(..., symlinks=False)` semantics / follow links) of: `commands/`, `skills/` (which includes the shared `.speckit/` scripts and `scripts/`), `settings.json`, and `CLAUDE.md`. Exclude secrets and large/irrelevant dirs.
- **Rationale**: Verified that `~/.claude/commands`, `~/.claude/skills`, `~/.claude/scripts`, `~/.claude/CLAUDE.md`, and `~/.claude/claude-powerline.json` are SYMLINKS into `~/profile/dotfiles/.claude/`, while `settings.json` is a real file. The links resolve on the host but would dangle inside a container, so contents must be copied with resolution (FR-008/C-009). The workflows reference other slash-commands and the `speckit-*` skills plus `.speckit` shared scripts, so `commands/` and `skills/` must be present for the workflow to run. The copy must be WRITABLE because Claude Code writes session/runtime state under HOME `.claude` during a `-p` run; a read-only mount would break it. Auth is provided via env, never copied from a file.
- **Exclusions (verified dirs to skip)**: `projects/`, `file-history/`, `history.jsonl`, `sessions/`, `session-env/`, `security/`, `telemetry/`, `shell-snapshots/`, `tasks/`, `backups/`, `plugins/`, `paste-cache/`, `stats-cache.json`, `stats-cache`, `debug/`, `ide/`, `mcp-needs-auth-cache.json`, and any `*.jsonl`/credential-like files. These are large, host-specific, or sensitive and are irrelevant to driving a workflow.
- **Risks**: settings.json may reference host MCP servers or hooks that cannot run in the container. Mitigation: the BENCHMARK_DIRECTIVE instructs the workflow to operate offline in `/app`; missing MCP servers degrade gracefully (Claude Code skips unreachable servers). If a hook hard-fails, the task is recorded as `errored`, not a silent pass.
- **Alternatives considered**: (a) Bind-mount `~/.claude` directly read-only - rejected; dangling symlinks inside the container and a read-only HOME breaks Claude Code's state writes. (b) Copy the whole tree - rejected; pulls in secrets and gigabytes of history/projects.

## R5. Headless workflow invocation and prompt construction (FR-006, FR-007)

- **Decision**: Drive the workflow with `claude -p` (headless print mode) inside the task container:
  `claude -p <prompt> --permission-mode bypassPermissions --output-format json --model <model> --append-system-prompt "<BENCHMARK_DIRECTIVE>"`, run from cwd `/app` with `HOME` set to the container HOME whose `.claude` is the mounted config. The task images run as root, so `HOME=/root` and the config mounts at `/root/.claude`; the harness sets `HOME` explicitly on the agent `docker exec` rather than assuming the image default, so `claude` reliably discovers commands/skills/settings. The prompt is written to a file inside the container (`/work/prompt.txt`); its FIRST line is the slash-command token (e.g. `/somecode`), followed by a blank line and the verbatim contents of the task's `instruction.md`. For a skill reference, the same convention applies (the leading token is the skill/slash name the owner uses locally).
- **Rationale**: Putting the slash-command at the very start is required for Claude Code to recognize and expand it. Feeding the prompt from the file (preferably piped to `claude -p` on stdin, or via an argument whose quoting is verified, rather than a bare `"$(cat /work/prompt.txt)"` that the shell re-interprets) avoids mangling multi-KB instructions that contain backticks, quotes, `$`, and newlines. `--output-format json` yields a single JSON object on stdout carrying result text plus token/cost usage (needed for FR-019). `--permission-mode bypassPermissions` lets the workflow edit files and run tools non-interactively (FR-007). `--model` enforces the owner-specified model on every attempt (FR-010); it overrides any `model:` frontmatter in the workflow file.
- **Alternatives considered**: (a) Pass the prompt as a CLI arg directly - rejected; quoting hazards with large instruction text. (b) `--output-format text` - rejected; loses structured token/cost data. (c) Interactive mode - rejected; violates the headless requirement.

## R6. Neutralization via `--append-system-prompt` only (C-010, FR-009)

- **Decision**: Inject a fixed BENCHMARK_DIRECTIVE through `--append-system-prompt`. It instructs the workflow that it runs in an isolated OFFLINE sandbox solving ONE task in `/app`, with NO git remote, NO GitHub, NO CI, and NO human to prompt; it MUST implement the change in place and MAY commit locally, but MUST NOT create worktrees or branches, MUST NOT push, MUST NOT open or manage pull requests, MUST NOT spawn sub-agents for PR/CI/review phases, MUST NOT monitor CI, and MUST skip any specify/clarify/PR/CI/QA phases that assume a connected environment; it ends the run as soon as the implementation is complete. As defense-in-depth it also tells the workflow to work solely from `/app` and the task instruction and not to seek out or read any held-out grading tests (these are physically absent during the agent phase per R8; this clause is belt-and-suspenders, not the isolation mechanism). The owner's workflow files are never edited.
- **Rationale**: Verified that both `somecode.md` and `story-to-live.md` create worktrees (via the `worktree` skill / `/worktree`), branch from main, push, open PRs with `gh`, and monitor CI across multiple spawned sub-agents - all meaningless and failing in an offline single-task sandbox with no remote/forge credentials. Steering via the appended system prompt satisfies the neutralization requirement without touching the read-only/owned workflow files (C-010). The directive text lives as a constant in `config.py`.
- **Risks**: A workflow might still attempt a forbidden step despite the directive. Mitigation: the sandbox simply has no remote and no `gh` auth, so such steps fail locally and are captured in `agent.err`; the run still terminates and the grader scores whatever change exists in `/app` (SC-007). The two-phase network model (R7) also guarantees the grader runs offline regardless.
- **Alternatives considered**: (a) Pre-edit copies of the workflow files to strip PR/CI phases - rejected by C-010 (no editing owner files) and brittle. (b) A git wrapper that no-ops push/pr inside the container - heavier and still would not stop the workflow's phase logic; the directive plus an absent remote is sufficient.

## R7. Two-phase single-container network model (C-008, FR-013, SC-008)

- **Decision**: Use ONE container per task with two phases. Agent phase: container has the default bridge network (outbound allowed) so `claude` can reach the model API. Grading phase: `docker network disconnect bridge <cid>` to take the SAME container offline, then run the verifier. The container is removed after grading.
- **Rationale**: The working tree the agent produced must be exactly what the verifier grades; using one container keeps `/app` in place between phases (FR-012). Disconnecting the network before grading guarantees the verifier is offline, matching the corpus's `allow_internet = false` requirement (all 113 tasks) and SC-008, even though the task image itself is run with a live network during the agent phase. Verified that `tests/test.sh` operates entirely on local `/app` and `/tests` with no network need (it `git reset`s to the base commit, applies `test.patch`, runs `test.sh base` and `test.sh new`).
- **Risks**: If a task's build step needed network during grading it would fail - but the corpus is explicitly offline-graded, and dependencies are pre-fetched in the prebuilt image (verified: Dockerfiles run `go mod download` / install at build time), so grading is self-contained.
- **Alternatives considered**: (a) Two containers (agent in one, copy patch to a fresh offline one for grading) - rejected; more moving parts, must re-materialize the working tree, and risks subtle drift from the agent's actual tree. (b) Run agent with `--network none` and rely on a proxy - rejected; the agent genuinely needs the model API online. (c) iptables rules inside the container - rejected; `network disconnect` is simpler and uses docker primitives.

## R8. Verifier mechanics and reward extraction (FR-016, FR-017, FR-018)

- **Decision**: Do NOT make the task's `tests/` directory available during the agent phase. After the agent phase completes and the network is disconnected, inject the tests into the still-running container with `docker cp <task_dir>/tests/. <cid>:/tests`, then run `bash /tests/test.sh`. Before running, pre-create `/logs/verifier` and `/logs/artifacts` inside the container. Read the reward from `/logs/verifier/reward.txt`: `1` => pass, anything else (including missing or unreadable) => non-pass (FR-017). Copy `model.patch`, `verifier.log`, and `reward.txt` out to the host run dir.
- **Rationale**: Verified by reading an actual `tests/test.sh`. It `cd /app`, captures the agent's diff vs the base commit to `/logs/artifacts/model.patch` (it DOES `mkdir -p` the artifacts dir itself), resets test-patch files to base by reading `/tests/test.patch`, applies the hidden `/tests/test.patch` with `git apply`, runs `bash /app/test.sh base` and `bash /app/test.sh new`, and writes `echo 1 > /logs/verifier/reward.txt` only when BOTH pass, else `echo 0`. Two facts ground the injection mechanism: (1) `test.sh` references the hidden tests strictly by the absolute path `/tests/test.patch`, so placing them at `/tests` immediately before running `test.sh` is sufficient - they need not be present any earlier; (2) `test.sh` writes to `/logs/verifier/reward.txt` WITHOUT creating `/logs/verifier`, so the harness MUST pre-create it or the write fails (exit 5). The verifier requires the base commit to be present in the repo and the agent's change to be in the working tree (it does `git add -A` for the artifact capture), which the single-container model preserves. Because the hidden tests live only in `tests/test.patch` and are injected via `docker cp` only after the agent phase, the agent never sees them (FR-005/FR-018), and `docker cp` goes through the docker daemon on the host, so it works even after the container's network is disconnected (it does not need container network).
- **Hidden-test isolation (R-RISK-3, resolved by physical isolation)**: FR-005/FR-018 require a HARD guarantee that the workflow (and any sub-agent it spawns) cannot read the held-out tests. A directive forbidding reads is too weak - the workflow could ignore it and read `/tests/test.patch` to game the result. The design therefore makes the hidden tests physically absent during the agent phase: `tests/` is never mounted and never copied in at container creation. The container during the agent phase holds ONLY `/app` (the repo at the base commit), the runtime mount, the writable `.claude` config mount, `/logs`, and the prompt under `/work`. Docker cannot add a bind mount to a running container, but it CAN copy files into one: after the agent phase finishes and the network is disconnected, the grader runs `docker cp <task_dir>/tests/. <cid>:/tests` (daemon-side copy, no container network needed) to materialize `/tests` for the first time, then runs `test.sh`. This preserves the single-container model (the working tree stays in place, so `model.patch` still captures the agent's exact change) while guaranteeing the tests do not exist while the agent runs. The BENCHMARK_DIRECTIVE still tells the workflow to operate only in `/app`, but that is defense-in-depth, NOT the integrity mechanism - integrity comes from the tests being physically absent.
- **Alternatives considered**: (a) Parse verifier stdout for a reward - rejected; the file is the authoritative contract. (b) Treat a non-zero `test.sh` exit as fail - insufficient; the reward file is the source of truth and distinguishes harness/verifier crashes (errored) from honest test failures (failed).

## R9. Deterministic subset selection (FR-004, NFR-001, SC-002)

- **Decision**: Discover task ids by listing immediate subdirectories of `tasks/` that contain a `task.toml`, sort the ids lexicographically for a stable canonical order, then select with `random.Random(seed)` using `sample(sorted_ids, k=min(n, len(ids)))`. Explicit id lists bypass sampling. Document the exact algorithm so it is reproducible on any machine. Parity with Pier's selection is explicitly NOT required - only OUR own reproducibility.
- **Rationale**: Sorting before seeding makes the candidate order independent of filesystem enumeration order, so the same `(n, seed)` yields identical ids on every machine (SC-002, NFR-001). `random.Random` is deterministic and stdlib. Capping at corpus size handles the "subset larger than corpus" edge case (spec edge cases) without error.
- **Alternatives considered**: (a) Mirror Pier's exact sampling - rejected; we do not depend on Pier (C-001) and the spec only requires our own reproducibility. (b) Hash-based selection - unnecessary complexity; seeded `random.sample` over sorted ids is sufficient and transparent.

## R10. Outcome classification, pass-rate math, and comparison (FR-020, FR-021)

- **Decision**: An `Outcome` enum with `PASSED`, `FAILED`, `ERRORED`, `NOT_ATTEMPTED`. `attempted = passed + failed + errored`; headline `pass_rate = passed / attempted` (0 when `attempted == 0`). Report `selected`, `attempted`, and `not_attempted` for reconciliation. For comparisons, compute each workflow's own pass rate over its attempted set AND a `common_attempted` set = intersection of task ids every workflow attempted; the head-to-head ranking uses common-attempted pass rates.
- **Rationale**: Directly encodes FR-020 and FR-021 and the spec's clarifications. Mapping: verifier reward 1 => PASSED; reward 0 or other present value => FAILED; agent timeout / agent crash / verifier crash / missing reward => ERRORED; image unobtainable or container failed to start => NOT_ATTEMPTED. This pure logic is the primary unit-test target (SC-006).
- **Alternatives considered**: (a) Booleans pass/fail only - rejected; cannot represent errored vs not-attempted, which the spec requires distinct. (b) Pass rate over `selected` as headline - rejected; the clarification fixed `attempted` as the denominator with full reconciliation.

## R11. Output layout, run identity, and reporting (FR-019, FR-022, FR-023, FR-024)

- **Decision**: Write everything under `jobs/<run-id>/` where `run-id = <UTC ISO-ish timestamp>-<short random suffix>` (e.g. `20260607T013455Z-a1b2c3`). Per-task artifacts nest at `jobs/<run-id>/tasks/<task-id>/<workflow-slug>/`. Emit `run.json` + `report.md` for single runs and `comparison.json` + `comparison.md` (with a per-task outcome matrix) for comparisons. Print an at-a-glance summary at the end (pass rate + path to artifacts).
- **Rationale**: `jobs` is already in `.gitignore` (verified), satisfying C-011/FR-024/SC-009 with no new ignored dir. The timestamp+suffix run id guarantees runs never overwrite each other and any run is locatable by id (FR-024). Dual JSON+Markdown satisfies FR-023. Nesting workflow under task keeps single-run and comparison layouts identical and collision-free.
- **Alternatives considered**: (a) A new top-level outputs dir - rejected; would need a new gitignore entry; reuse the existing `jobs` ignore. (b) Markdown only - rejected; FR-023 requires machine-readable too.

## R12. Preflight validation order (FR-010, FR-025, NFR-006, SC-005)

- **Decision**: Before provisioning ANYTHING (no runtime build, no config copy, no container), validate in order: (1) exactly one credential present - `ANTHROPIC_API_KEY` or `CLAUDE_CODE_OAUTH_TOKEN`; if neither, error naming both accepted forms and how to mint a token (`claude setup-token`); (2) a `--model` was provided; (3) `docker` is available and the daemon responds. Any failure aborts with a single clear actionable message and zero side effects.
- **Rationale**: Verified neither credential is currently set in the environment, confirming this guard's relevance. Failing before provisioning prevents wasting runtime-build and image-pull work on a misconfiguration (NFR-006/SC-005). Accepting exactly one of the two credential forms matches the clarification and FR-025.
- **Alternatives considered**: (a) Validate lazily on first container - rejected; wastes provisioning and violates "zero sandboxes provisioned" (SC-005). (b) Accept ambient host login - rejected by C-006 (no ambient credential usable inside the isolated environment).

## R13. Credential delivery into the container (C-006)

- **Decision**: Pass the validated credential into each task container as an environment variable on `docker run` (`-e ANTHROPIC_API_KEY` or `-e CLAUDE_CODE_OAUTH_TOKEN`), never written to a file or baked into an image. The harness reads it from its own environment.
- **Rationale**: Env injection keeps the secret out of any persisted artifact and out of the copied config (R4). Claude Code reads these standard env vars for auth. Aligns with "secrets MUST NOT be committed" and the spec's "auth passed via env, never from a file".
- **Alternatives considered**: (a) Write the token into the mounted `.claude` - rejected; risks the secret landing in run artifacts. (b) `docker login`-style credential helper - not applicable to claude-code auth.

## R14. Timeouts and resource limits (FR-011, FR-014)

- **Decision**: Apply `--cpus` and `--memory` from each `task.toml` `[environment]` (verified uniformly `cpus = 2`, `memory_mb = 8192`, parsed per task to stay correct if any differ). Enforce the agent phase with a host-side `subprocess` timeout equal to `[agent].timeout_sec` and the grading phase with a host-side timeout equal to `[verifier].timeout_sec`. A timeout kills the `docker exec` and records the task ERRORED with a timeout reason; whatever partial change exists is still captured by the grader if grading can run, else the partial state is noted.
- **Rationale**: Matches FR-014 (separate budgets for agent vs grading) and FR-011 (use task-specified environment/limits). Host-side timeouts around `docker exec` are reliable and do not require the container to self-limit. Verified the toml fields exist: `[agent] timeout_sec` and `[verifier] timeout_sec` (e.g. 5400 and 1800 in the sampled task).
- **Alternatives considered**: (a) `timeout(1)` inside the container - rejected; not guaranteed present in every image and harder to observe from the host. (b) docker `--stop-timeout` - applies to stop, not exec duration.

## Summary of residual risks (carried into tasks.md and analyze)

- **R-RISK-1 glibc/musl**: runtime assumes glibc task images; musl images would break the mounted runtime. Detect via a `claude --version` smoke check on first use; record `errored` on failure.
- **R-RISK-2 pinned versions**: claude-code and node versions are pinned constants; if a pinned version is unavailable the runtime build fails fast with a clear error.
- **R-RISK-3 hidden-test exposure (mitigated)**: The held-out tests are physically absent during the agent phase - `tests/` is never mounted and is injected with `docker cp` only after the network is disconnected, immediately before grading. This is a hard guarantee for FR-005/FR-018, not a directive. The single-container model is preserved (working tree stays in place for `model.patch` capture). The BENCHMARK_DIRECTIVE remains as defense-in-depth only. Residual: none for hidden-test reads; the only assumption is that `docker cp` into a running, network-disconnected container succeeds (verified: `docker cp` is a daemon-side host operation, independent of container networking).
- **R-RISK-4 MCP/hooks in copied settings**: host-specific MCP servers/hooks referenced by `settings.json` may be unreachable in the container; they degrade gracefully, and a hard hook failure yields `errored`, never a silent pass.
- **R-RISK-5 model usage cost/time**: real model-driven runs cost money and wall-clock proportional to subset size; the owner accepts this (spec Assumptions). Preflight does not estimate cost.
